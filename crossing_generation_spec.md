# Crossing Generation Procedure (codebase‑agnostic spec)

> **Goal:** Given a street network and derived sidewalk geometry, produce **crossing polylines** (A‑B‑C‑D‑E) and **kerb points** at eligible intersections. The method is intentionally codebase‑agnostic so it can be plugged into any GIS stack.

---

## 1) High‑level idea

* Build/receive a **dissolved sidewalk geometry** (multipart or unioned) from the sidewalk‑buffer step.
* For each split street segment, identify intersection endpoints that are likely to host crossings.
* For each eligible endpoint, place a **crossing centerpoint** at an specified distance **inward** along the street, compute a **perpendicular direction vector**, and **cast rays** in both directions to find where the crossing meets the sidewalks.
* If a crossing is **too long**, push the centerpoint further inward and retry, up to a cap.
* Emit the crossing line and kerb points with basic attributes (length, flags).

This reflects how SidewalKreator’s workflow handles “crossings and kerbs generation” as a distinct stage in the overall pipeline, separate from sidewalk geometry creation and export. ([European Journal of Geography][1])

---

## 2) Inputs

* **Street segments (polyline layer)** split at intersections; each feature includes an estimated **width**.
* **Dissolved sidewalks (single geometry)** for intersection testing (union/multipart of all sidewalk polygons or outlines).
* **Parameters**

  * `inward_offset`: meters to move from the intersection into the segment before searching.
  * `extra_length`: meters to add to the initial vector length (road width + extra).
  * `length_tolerance_pct`: percent tolerance above (road width + extra) allowed for crossings.
  * `curve_radius`: meters used in prior buffering that mildly affects inward placement.
  * `max_iterations`: cap on retry loops when a crossing is too long.
  * `abs_max_crossing_len`: hard ceiling on any crossing length.
  * `kerbs_percent`: how far from the endpoints the kerbs shall be placed

*(Exact names/units can be adapted to your environment.)*

---

## 3) Outputs

* **Crossing polylines** with ordered points **A–B–C–D–E**

  * **A** and **E**: intersection with sidewalks on each side.
  * **C**: midpoint.
  * **B** and **D**: near‑kerb points found by simple interpolation along **A→C** and **E→C**.
* **Attributes (suggested)**

  * `length_m`, `length_ok` (boolean), `nearest_centerpoint` (ID), `above_tolerance` (boolean).
* **kerb point features** at **B** and **D**.

---

## 4) Procedure (step‑by‑step)

### 4.1 Prepare geometry

1. Ensure a **dissolved sidewalk geometry** is available (union or multipart).
2. (Optional) Prepare the geometry for **fast intersection** (e.g., prepared/STR‑tree).
   *This mirrors how the plugin prepares a single dissolved geometry to accelerate intersection tests.* ([QGIS Plugins][2])

### 4.2 Find candidate endpoints on streets

3. For each **street segment** `S`, get its **first** (`P0`) and **last** (`PF`) vertex.
4. For each of `P0` and `PF`, query a spatial index of street segments within a small buffer (e.g., ~0.1 m) to **count** how many segments intersect that vicinity.

   * **Eligible** endpoints typically have **degree > 2** (i.e., real intersections), not simple mid‑block splits.

### 4.3 Place a centerpoint and direction vector

5. Compute an **inward distance** from the endpoint along `S`:

   * `inward = 0.5 * (major_intersecting_width) + curve_radius + inward_offset`
   * If `inward > 0.5 * length(S)`, clamp to a fraction of `length(S)`.
6. Interpolate the **centerpoint** `C` at distance `inward` along `S`.
7. Compute the **crossing direction**:

   * Default: **perpendicular** to `S` at `C`.
   * Alternative: **parallel** to an adjacent major street (option/flag).
8. Set an **initial vector length** `L0 = street_width(S) + extra_length`.

> The plugin’s academic/talk description explicitly notes that crossings are generated by **growing a vector iteratively in a perpendicular (or parallel) direction** until it intersects the dissolved sidewalks, with filters for too‑long or too‑close crossings. ([Pretalx][3])

### 4.4 Find sidewalk intersections (two‑sided ray casting)

9. From `C`, cast two rays along `+V` and `−V` (where `|V| = L0`), intersecting with the **dissolved sidewalks**.
10. If a side **doesn’t intersect**, **scale** that side’s vector (e.g., ×2) and retry (repeat as needed or up to a cap).
11. When both intersections are found, call them **A** and **E** (order them consistently).

### 4.5 Length control and inward adjustment

12. Form the provisional crossing line `AE` and compute `len(AE)`.
13. If `len(AE) > abs_max_crossing_len`, **discard** this crossing.
14. Compute `max_allowed = (street_width(S) + extra_length) * (1 + length_tolerance_pct/100)`.
15. If `len(AE) > max_allowed` and the **centerpoint can go further inward**:

    * Increase `inward` by a small increment; recompute `C`, re‑cast rays (steps 9–11).
    * Stop when `inward ≥ 0.5*length(S)` or `max_iterations` reached.

### 4.6 Build final crossing geometry and attributes

16. Set **C = midpoint** of `A` and `E`.
17. Interpolate **B** on segment `AC` (e.g., 30% from `A`), and **D** on `EC`.
18. Save the polyline **A‑B‑C‑D‑E** and copy kerb points **B** and **D** with attributes: `length_m`, `length_ok`, `above_tolerance`, etc.

## 5) Pseudocode (drop‑in logic)

```pseudo
function generate_crossings(streets, dissolved_sidewalks, params):
  out_lines = []
  out_kerbs = []

  for each segment S in streets:
    for endpoint in [start(S), end(S)]:
      if degree(endpoint) <= 2:
        continue

      inward = compute_inward_distance(S, endpoint, params)
      inward = clamp(inward, 0, 0.5 * length(S))
      C = interpolate(S, distance_from_endpoint=SenseAware(inward, endpoint))

      V = direction_vector(S, mode=params.direction_mode, at=C)
      L0 = width(S) + params.extra_length

      (A, E) = cast_two_sided_rays(C, V, L0, dissolved_sidewalks, params)
      if A == null or E == null:
        continue

      if length(A,E) > params.abs_max_crossing_len:
        continue

      max_allowed = (width(S) + params.extra_length) * (1 + params.length_tolerance_pct/100)
      tries = 0
      while length(A,E) > max_allowed and tries < params.max_iterations and inward < 0.5*length(S):
        inward += params.inward_increment
        C = interpolate_inward(S, endpoint, inward)
        (A, E) = cast_two_sided_rays(C, V, L0, dissolved_sidewalks, params)
        if A == null or E == null:
          break
        tries += 1
      if A == null or E == null:
        continue

      C = midpoint(A,E)
      B = interpolate(A, C, t=params.kerb_interp)  # e.g., t = 0.3
      D = interpolate(E, C, t=params.kerb_interp)
      out_lines.append(polyline(A,B,C,D,E, attrs(...)))
      out_kerbs.extend([B, D])

  return out_lines, out_kerbs
```

---

## 6) Integration notes

* **Geometry engine:** Any robust library (QGIS, GEOS/Shapely, JTS). You need: buffer/union, interpolate along line, line–polygon intersections, and spatial indexing.
* **Where to plug:** Insert after your **sidewalk‑buffer/dissolve** stage and before **export**. That mirrors SidewalKreator’s pipeline stages (data fetch → sidewalk generation → crossings/kerbs → optional sidewalk splitting → export). ([European Journal of Geography][1])
* **Data model assumption:** Streets are OSM‑like `highway=*` lines; sidewalks are generated geometrically from those lines (typical in SidewalKreator and derivative workflows). ([OpenStreetMap][4])

---

## 7) Behavior & safeguards

* **Intersection eligibility:** Use degree (>2) to avoid placing crossings mid‑block.
* **Fail‑safe:** Cap crossing length and iteration count; discard outliers.
* **Robustness:** Sidewalk missing on one side → ray won’t intersect → crossing skipped.
* **User control knobs:** `inward_offset`, `extra_length`, `length_tolerance_pct`, `max_iterations`, `abs_max_crossing_len`.

---

## 8) Why this design?

* Keeps the logic **stateless and pluggable** (no reliance on a specific plugin API).
* Uses a **ray‑cast + retry** approach that is documented in the SidewalKreator method (vector grows in perpendicular/parallel direction until sidewalk intersection), with simple filters to avoid bad crossings. ([Pretalx][3])
* Matches published descriptions of the tool’s workflow and features (crossing and kerb generation as a dedicated stage), so it’s easy to reason about and port. ([QGIS Plugins][2])
